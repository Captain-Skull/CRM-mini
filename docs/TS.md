**Версия:** 0.9

## 1. Введение

### 1.1. Цель проекта
Разработать минимально жизнеспособный продукт (MVP) для автоматизации учёта лидов и построения воронки продаж. Система должна позволять фиксировать лиды, отслеживать их статусы, рассчитывать ключевые показатели эффективности (KPI) и визуализировать процесс на канбан-доске.

### 1.2. Границы проекта (Scope)
**В рамках MVP реализуются:**
- Ведение справочников: «Лиды», «Источники».
- Фиксация истории изменения статусов лидов.
- Импорт данных из CSV-файлов.
- Отчёт «Воронка продаж» с фильтрами и расчётом KPI.
- Прототип пользовательского интерфейса (канбан-доска, карточка лида).
- Мок-API для демонстрации интеграции.

**Вне рамок MVP:**
- Интеграция с реальными внешними системами (телефония, почта).
- Работа с персональными данными (используется только синтетика).
- Полноценная система прав доступа.

## 2. Термины и определения

| Термин | Определение |
|--------|-------------|
| **Лид** | Потенциальный клиент, оставивший контактную информацию. |
| **Статус лида** | Этап обработки лида: `new` (`новый`), `qualified` (`квалифицированный`), `proposal` (`предложение`), `won` (`успешно`), `lost` (`отказ`). |
| **Источник** | Канал привлечения лида (сайт, реклама, рекомендация и т.п.). |
| **Воронка продаж** | Отчёт, показывающий количество лидов на каждом этапе и конверсию между этапами. |
| **KPI** | Ключевые показатели эффективности (конверсия, средняя длительность этапов). |

## 3. Функциональные требования

### 3.1. Управление лидами

| ID | Требование | Приоритет (MoSCoW) |
|----|------------|---------------------|
| FR-01 | Система должна обеспечивать создание и хранение лидов со следующими атрибутами: уникальный идентификатор, имя, фамилия, источник, ответственный менеджер, время создания, время попадания в текущий статус, текущий статус. | Must |
| FR-02 | Система должна автоматически фиксировать дату и время каждого изменения статуса лида в истории. | Must |
| FR-03 | Должна быть реализована возможность импорта лидов и истории статусов из CSV-файлов. Импорт должен быть идемпотентным (повторная загрузка не создаёт дубликатов). | Must |
| FR-04 | Пользователь должен иметь возможность просматривать карточку лида с полной информацией и историей статусов. | Should |
| FR-05 | В интерфейсе должна быть реализована канбан-доска, отображающая лиды в колонках по статусам. | Must |

### 3.2. Отчётность и KPI

| ID | Требование | Приоритет |
|----|------------|-----------|
| FR-06 | Отчёт «Воронка продаж» должен показывать количество лидов на каждом этапе, а также конверсию между этапами (например, `qualified → proposal`). | Must |
| FR-07 | Отчёт должен поддерживать фильтрацию по периоду, источнику и ответственному менеджеру. | Should |
| FR-08 | Система должна рассчитывать следующие KPI:<br> - Конверсия `new → won` (в процентах).<br> - Конверсия между каждым этапом (в процентах).<br> - Средняя длительность этапов (по времени). | Must |
| FR-09 | KPI должны отображаться высчитываться при формировании отчета. | Must |
| FR-10 | Должна быть предусмотрена возможность экспорта отчёта в CSV. | Could |

### 3.3. Валидация и бизнес-правила

| ID | Требование | Приоритет |
|----|------------|-----------|
| FR-11 | Статусы лидов могут изменяться только в допустимой последовательности: `new` → `qualified` → `proposal` → `won`/`lost`. Обратные переходы запрещены (кроме административных корректировок, которые в MVP не реализуются). | Must |
| FR-12 | Для перехода в статус `lost` обязательно указание причины отказа (поле `reason` в истории). | Could |
| FR-13 | При импорте данных система должна проверять ссылочную целостность (например, что у записи истории есть соответствующий лид). | Must |

## 4. Требования к данным

### 4.2. Форматы файлов импорта

- **leads.csv** (поля: id, name, source_id, owner_id, created_at, current_stage)
- **lead_stages.csv** (поля: id, lead_id, stage, entered_at, left_at, reason)

Формат CSV: разделитель — запятая, строки в двойных кавычках, кодировка UTF-8.

### 4.3. Валидации данных

- `entered_at` ≤ `left_at` (если `left_at` не пусто).
- Для каждого лида записи в `lead_stages` должны идти в хронологическом порядке без пропусков статусов.
- Поле `stage` должно принимать только значения из перечисления (new, qualified, proposal, won, lost) (новый, квалифицированный, предложение, успешно, отказ).
- При импорте проверять наличие связанных записей (source_id должен существовать в справочнике источников).

## 5. Требования к пользовательскому интерфейсу

| ID | Требование |
|----|------------|
| UI-01 | Канбан-доска должна содержать колонки для каждого статуса (`new`, `qualified`, `proposal`, `won`, `lost`). |
| UI-02 | В каждой колонке отображаются карточки лидов (минимум: имя, фамилия). |
| UI-03 | Должна быть возможность «перетащить» карточку из одной колонки в другую. |
| UI-04 | При клике на карточку открывается детальная информация о лиде и история статусов. |
| UI-05 | Страница со списком всех лидов и со всем их данными (id, имя, фамилия, стадия, источник, менеджер, создан, попалВСтатус). |
| UI-06 | Отчёт «Воронка» должен быть представлен в виде таблицы и/или диаграммы. |
| UI-07 | В отчете должны присутствовать все KPI (конверсия и средняя длительность). |
| UI-08 | Должны быть реализованы конпки экспорта и иморта данных. |

## 6. Требования к API

| ID | Требование |
|----|------------|
| IF-01 | Должен быть реализован мок-сервер, эмулирующий REST API с эндпоинтами: <br> - `GET /leads` — возвращает список лидов (поддержка фильтрации по статусу).<br> - `GET /leads/{id}` — возвращает детальную информацию о лиде.<br> - `POST /leads` — создаёт нового лида. |

## 7. Нефункциональные требования

| ID | Требование |
|----|------------|
| NFR-01 | Система должна корректно обрабатывать не менее 1000 лидов и 3000 записей истории без существенной потери производительности. |
| NFR-02 | Импорт данных должен быть идемпотентным: повторная загрузка одного и того же файла не должна создавать дубликаты. |

| NFR-04 | Код должен храниться в Git-репозитории, вестись история изменений, Pull Request'ы проходить ревью. |
| NFR-05 | Документация (ТЗ, модели, инструкции) должна быть в папке `/docs`. |
